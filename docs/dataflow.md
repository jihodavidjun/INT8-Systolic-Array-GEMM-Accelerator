# Dataflow

This project implements an INT8 GEMM-style datapath using a systolic array. The key design goal is **streaming**: feed A/B as sequences, let data “flow” through PEs, and accumulate partial sums locally.

---

## High-level pipeline

At a high level, the system is:

**Input vectors/files → FIFOs → controller → systolic array → output C stream**

- **FIFOs** decouple testbench / memory source from the array’s consumption rate.
- **Controller** determines when the array is enabled and when outputs are valid.
- **Systolic array** performs multiply-accumulate (MAC) in each PE while forwarding A and B.

---

## Compute model (GEMM mapping)

We target a matrix multiply of the form:

- `C = A × B`
- A and B are INT8 (signed)
- C is ACC_W (default 32-bit signed accumulator)

For a 4×4 systolic array:
- `A` is 4×K
- `B` is K×4
- `C` is 4×4

The array runs for `K` cycles of MAC (plus a small fill/drain latency due to systolic forwarding).

---

## Streaming schedule (intuition)

Each cycle:
- Row inputs supply `a_in[row]`
- Column inputs supply `b_in[col]`
- Each PE consumes its local `(a, b)` pair, updates accumulator, and forwards:
  - `a` to the right
  - `b` downward

So values of A move east, values of B move south.

---

## Data movement diagram (4×4)

A stream enters from the left (rows), B stream enters from the top (cols).

```
      b00  b01  b02   b03
       ↓   ↓    ↓     ↓
a00 → PE → PE → PE → PE
       ↓   ↓    ↓     ↓
a10 → PE → PE → PE → PE
       ↓   ↓    ↓     ↓
a20 → PE → PE → PE → PE
       ↓   ↓    ↓     ↓
a30 → PE → PE → PE → PE
```

Each PE holds one accumulator `acc_out[row][col]`.

## Top-level integration

The “streaming” top is implemented in:

- `rtl/top.sv` (integration)
- `rtl/fifo.sv` (input buffering)
- `rtl/controller.sv` (control of enable/valid)
- `rtl/sa4x4.sv` (array)
- `rtl/pe.sv` (PE MAC + forwarding regs)

### Input FIFOs

- A FIFO holds incoming A words (packed stream)
- B FIFO holds incoming B words (packed stream)
- The controller pops when it is safe to advance the array

This prevents the array from stalling the testbench and makes the interface closer to a real accelerator.

---

## Output format

The array produces a 4×4 block of `C` values.
In streaming mode, results are typically emitted in a deterministic order (e.g., row-major).

The expected output for verification is generated by Python and stored in:
- `data/exp_c.memh`

---

## Notes on parameterization

A parametric NxN module exists (`rtl/saNxN.sv`) to demonstrate scalability, but correctness/timing is validated primarily on a fixed 4×4 instance (simplifies debug and matches most tests).
